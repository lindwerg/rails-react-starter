## 1. Stack overview

**Monorepo** with two apps:

| Layer | Tech |
|---|---|
| Backend | Ruby 3.3 + Rails 8 (API mode), PostgreSQL 16, Solid Queue/Cache/Cable |
| Backend arch | Modular monolith via **Packwerk** (4 layers: orchestrator → business_domain → platform → utility) |
| Frontend | Node 22 + pnpm + Vite 6 + React 19 + TypeScript 5.6 (strict) |
| Frontend arch | **Feature-Sliced Design (FSD)** — 6 layers: app/pages/widgets/features/entities/shared |
| API contract | OpenAPI generated by `rswag`, types regenerated on the frontend via `openapi-typescript` |
| Auth | JWT in **httpOnly cookie** (never localStorage) |
| Tests | RSpec, FactoryBot, Vitest, Testing Library, Playwright, MSW |
| Lint | RuboCop, Brakeman, ESLint 9 (FSD plugin), Prettier, lefthook pre-commit |
| Deploy | Kamal 2 → Hetzner/DigitalOcean |
| CI | GitHub Actions (3 workflows) |

## 2. The TDD rule (non-negotiable)

**Every new feature follows Red → Green → Refactor.** Tests before code.

| Layer | Required test types |
|---|---|
| Backend model | model spec (validations, associations, scopes) |
| Backend service | service spec (success path, failure path, edge cases) |
| Backend endpoint | request spec (status, shape, auth) |
| Frontend component | Vitest + Testing Library (renders, interaction, a11y) |
| Frontend feature | integration test (form submit, mutation, error state) |
| Critical user flow | Playwright E2E |

CI rejects any PR where:
- Tests are failing
- New code lowers coverage below threshold
- A new public function/component has no test
- `bin/packwerk check` shows boundary violations

**You don't get to skip tests "because it's just a small change."** No tests = no merge.

## 3. Architectural rules

### Backend — Packwerk layers

```
orchestrator      ← presentation (controllers, serializers)  — pack: api
business_domain   ← business logic                            — packs: auth, users, posts, ...
platform          ← infrastructure                            — pack: platform
utility           ← pure utilities                            — pack: shared
```

A pack can only depend on packs **at the same layer or below**. Adding a new domain = new pack in `business_domain` layer. **Do not put domain logic in `app/`** — `app/` is for Rails-required boilerplate (config, application_controller, etc.).

Each pack:
```
packs/<name>/
├── package.yml          # enforce_dependencies: true, enforce_privacy: true, layer: ...
├── app/
│   ├── public/          # explicit public API of the pack (constants visible to other packs)
│   ├── models/
│   ├── services/
│   ├── queries/
│   └── forms/
├── spec/
│   ├── factories/
│   └── ...
└── README.md            # what this pack is responsible for
```

Always run `bin/packwerk check` before committing backend changes.

### Frontend — FSD layers

```
app          ← providers, router, store, styles
pages        ← page components (route entries)
widgets      ← composite blocks reused across pages
features     ← user actions that produce business value
entities     ← business entities (User, Post, Session)
shared       ← UI kit, api client, libs (no business knowledge)
```

**Import rules** (enforced by `eslint-plugin-boundaries` + `@feature-sliced/eslint-config`):
- A layer can import only from layers **below** it.
- A slice **cannot import from a sibling slice on the same layer** — go through `entities` or `shared`.
- Public API: each slice exposes only what's in its `index.ts`. **No deep imports.**
- **No relative imports across slices.** Use the `@/` alias.

Each slice:
```
features/auth-by-email/
├── ui/
├── model/
├── api/
├── lib/
└── index.ts             # public API
```

## 4. Working with Claude — workflow

### Always start with plan mode

For anything beyond a one-line fix:
1. User describes intent
2. Claude enters plan mode (`Shift+Tab` until "plan mode on"), gathers context, writes a plan to `~/.claude/plans/<slug>.md`
3. Claude calls `ExitPlanMode` for user approval
4. Claude executes
5. **Claude writes a log to `.claude/logs/`** (see §5)
6. Claude commits with conventional-commit message

### Reading context at session start

Whenever you (Claude) start in this repo, read in this order:
1. `CLAUDE.md` (this file)
2. `PROGRESS.md` (index of all logs)
3. The two latest files in `.claude/logs/` — they tell you what just happened
4. Open plan in `~/.claude/plans/` if user references one

Then propose next steps.

## 5. Progress logs (`.claude/logs/`)

**After every plan-mode execution**, create a file in `.claude/logs/` named:

```
YYYY-MM-DD-<short-slug>.md
```

Use this template (copied from `.claude/logs/_TEMPLATE.md`):

```markdown
---
date: YYYY-MM-DD
plan: <plan-file-name-or-"none">
status: completed | partial | rolled-back
commits:
  - <sha-prefix> <message>
---

# <Title>

## Что сделано
- bullet list of concrete changes

## Решения и почему
- decision → rationale (especially non-obvious tradeoffs)

## Открытые вопросы / TODO
- known limitations, deferred work

## Куда дальше
- what the next session should pick up
```

Then **append one line to `PROGRESS.md`** as the index entry. PROGRESS.md is what future Claude sessions read first to understand history.

`make log` scaffolds a new log file from the template.

## 6. Commands (run from repo root)

| Command | What it does |
|---|---|
| `make setup` | Install Ruby/Node/pnpm via mise, bundle, pnpm install, db:prepare, lefthook install |

| `make log` | Create new log file in `.claude/logs/` |

- [ ] `PROGRESS.md` updated
- [ ] Commit message follows conventional commits (`feat(scope): ...`, `fix(scope): ...`, `test(scope): ...`)

## 8. Naming conventions

- **Branches**: `feat/<slug>`, `fix/<slug>`, `chore/<slug>`
- **Commits**: `<type>(<scope>): <subject>` — `type` ∈ {feat, fix, refactor, test, docs, chore, perf, ci}
- **Backend packs**: lowercase, plural for entities (`posts`, not `post`)
- **Frontend slices**: kebab-case (`auth-by-email`, `create-post`)
- **DB columns**: snake_case
- **JSON keys on the wire**: camelCase (transformed by serializer/middleware)

---

## 9. Things you should NOT do

- ❌ Skip tests for "trivial" changes
- ❌ Put business logic in Rails `app/` directly (use packs)
- ❌ Import across FSD slices on the same layer
- ❌ Use deep imports — only `index.ts` of each slice
- ❌ Store JWT in localStorage
- ❌ Use `--no-verify` on commits
- ❌ Force-push to `main`
- ❌ Add a dependency without explaining why in the log
- ❌ Create files outside the documented architecture without updating CLAUDE.md first

## 10. When in doubt — verify with context7

For any library/framework question, **use the context7 MCP server** before answering. Examples:
- "How does Solid Queue retry?" → `mcp__context7__query-docs` on `/websites/guides_rubyonrails_v8_0`
- "Latest TanStack Query mutation API?" → resolve `TanStack Query` then query
- "Packwerk layer enforcement?" → query `/rubyatscale/packwerk-extensions`

This catches recent API changes that the model's training data may have missed.
