## 2. THE NON-NEGOTIABLES

### 2.1. MCP servers — when to use them

Two MCP servers are configured project-wide in `.mcp.json` and **must be used**:

#### **`context7`** — for ANY library/framework question
Triggered by:
- User asks "how does X work" where X is a library, framework, SDK, API, or CLI tool
- You need to write code using a library you haven't touched in this session
- You're debugging a library-specific error
- Migrating between versions
- Configuring a library

Workflow:
1. `mcp__context7__resolve-library-id` (libraryName, query)
2. `mcp__context7__query-docs` (libraryId, query)
3. Cite source in your response: *"per /rails/rails docs..."*

**Don't trust your training data alone.** Even for things you "know" (Rails, React, TanStack Query) — verify when in doubt. Faster than fixing a bug from outdated knowledge.

#### **`magic-mcp` (21st.dev)** — for ANY new UI component
Triggered by:
- User asks for a new component, layout, page section, form widget, modal, etc.
- You're about to write JSX in `frontend/src/shared/ui/**`, `frontend/src/widgets/**`, or a new `features/*/ui/`
- User says "make it look nice", "add a button", "redesign X"

Workflow:
1. `mcp__magic-mcp__21st_magic_component_inspiration` — see options
2. `mcp__magic-mcp__21st_magic_component_builder` — generate
3. Place generated component into the correct FSD slice (§4.2)
4. Hand-edit if needed; **always wrap with our `cn()` from `@/shared/lib`** for Tailwind merging
5. Add a Storybook story + a Vitest test (TDD §3) — this part is on you, not the MCP

**Don't write UI from scratch when magic-mcp can give you a starting point.** The exception is trivial elements (a single `<span>` wrapper, a one-line text fix). For anything composed of >1 primitive — use the MCP.

For **logos and brand icons** specifically: `mcp__magic-mcp__logo_search` before reaching for `lucide-react`.

If the MCP server is unavailable in the current session, say so out loud and ask the user how to proceed (do *not* silently write from scratch).

### 2.2. TDD is the law

Every new behavior follows **Red → Green → Refactor**.

#### Red
1. Write the test **first**.
2. Run it. Confirm it fails for the right reason (assertion fails, not "module not found").
3. **Commit the failing test** before writing implementation. Yes, really. The commit message is `test(scope): describe behavior` — body explains what the test enforces.

#### Green
4. Write the **minimum** code to make the test pass.
5. Run the test. Confirm green.
6. Run the relevant linter/typecheck on the changed files.

#### Refactor
7. Now you may improve naming, extract helpers, deduplicate. Tests stay green.
8. Commit `refactor(scope): ...` separately.

**Required test types per change:**

| Change | Required tests |
|---|---|
| New AR model | model spec (validations, associations, scopes, custom methods) |
| New service / job / mailer | service spec (success path + 2+ failure paths + edge cases) |
| New controller endpoint | request spec (status code + JSON shape + auth check + pundit check) |
| New policy | policy spec (allowed actions × actor matrix) |
| New React component (presentation) | Vitest + Testing Library (render + interaction + accessibility query) |
| New React feature (form / mutation) | integration test (RHF submit, mutation succeeds, mutation fails surfaces error) |
| New page or critical user flow | Playwright E2E |

**TDD violations CI will catch automatically:**
- `make pack-check` fails on Packwerk boundary violations
- `bin/rspec` fails on regression
- SimpleCov fails if coverage drops below 90% (backend) / 80% (frontend)

**You don't ship a feature with "tests later".** "Tests later" = "tests never". If the user begs you to skip — push back once, then comply with a TODO and a `chore(test):` issue noted in the log.

### 2.3. Architecture — Packwerk + FSD

Both layered. Higher layers may import from lower; lower may NEVER import from higher. Both enforced automatically.

#### Backend layers (top → bottom, defined in `backend/packwerk.yml`)
```
orchestrator      ← packs/api                — controllers, serializers, presentation only
business_domain   ← packs/auth, users, posts  — pure domain logic, models, services
platform          ← packs/platform            — jobs, mailers, storage, infrastructure
utility           ← packs/shared              — pure helpers (Result, Errors, ValueObjects)
```

**Rules:**
- A new domain → a new pack in `business_domain`. Never put domain code in `app/`.
- Cross-pack code lives in `app/public/` of the source pack (Packwerk privacy enforces it).
- Service objects use **Result pattern** (`Shared::Result.success / .failure(:code, msg)`). No raise-for-control-flow in business logic.
- Each pack has a `README.md` explaining its responsibility.

#### Frontend layers (top → bottom, enforced by `eslint-plugin-boundaries`)
```
app          ← providers, router, store, styles
pages        ← page components (route entries)
widgets      ← composite blocks (PostFeed, Header, ...)
features     ← user actions producing business value (auth-by-email, create-post, ...)
entities     ← business entities with thin API + UI (User, Post, Session)
shared       ← UI kit, api client, libs (no business knowledge)
```

**Rules:**
- A slice exposes only what's in its `index.ts`. **Deep imports are forbidden** and ESLint will reject them.
- A slice **cannot import a sibling on the same layer**. Go through `entities/` or `shared/`. (e.g. `features/comments` cannot `@/features/auth-by-email` — they're peers.)
- All cross-slice imports use `@/<layer>/<slice>` alias. **No relative `../../`.**
- New visual primitive → `shared/ui/<name>/`. New business UI → `entities/<name>/ui/` or its feature.
- Business state → TanStack Query (server state) or Zustand (client state). **No `useState` for cross-component data.**

If you're about to violate these — STOP. Restructure or open plan-mode to discuss.

### 2.4. Plan mode is mandatory for non-trivial work

**Trivial = one-line fix, typo, comment edit, rename in a single file.** Everything else goes through plan mode:

1. User describes intent.
2. Toggle plan mode (Shift+Tab) — do not make changes.
3. **Use context7 if any library is involved.**
4. Read the affected files. Understand current state.
5. Write the plan to `~/.claude/plans/<slug>.md` with sections:
   - **Context** — what's the change and why
   - **Approach** — concrete file list + what changes in each
   - **Tests** — list of specs/tests you will write FIRST
   - **Verification** — how to test end-to-end (commands)
6. Call `ExitPlanMode` for approval.
7. Execute. Write tests first (§2.2). Run checks (§2.5). Write log (§2.6). Commit.

Plans are throwaway after the work is done — but the **log** that follows is permanent.

### 2.5. Pre-PR / pre-commit checklist (Claude must verify before saying "done")

Run, in order:

```bash
make test          # RSpec + Vitest
make lint          # RuboCop + ESLint + Prettier check
make typecheck     # TypeScript strict
make security      # Brakeman + bundler-audit + npm audit
make pack-check    # Packwerk boundaries
```

All MUST be green. If any is red:
- Fix it before claiming done. Don't propose a follow-up "fix in next PR".
- If genuinely out of scope (pre-existing failure unrelated to your change) — call it out explicitly and ask how to handle.

**Do NOT use `--no-verify`, `--skip-checks`, or any flag that bypasses lefthook hooks.** Investigate root cause; never bypass.

### 2.6. Logs — automatic memory across sessions

After every meaningful work session (= after `ExitPlanMode` execution completes, or after a multi-step task):

1. Run `make log` (it scaffolds the file from `.claude/logs/_TEMPLATE.md`).
2. Fill all four sections: **Что сделано / Решения и почему / Открытые вопросы / Куда дальше**.
3. Add **one line** to `PROGRESS.md` at the top of the dated section.
4. Commit log + PROGRESS.md update along with the work (single commit, not separate).

Log is the only thing future-you (next session) will see. Make it good.

**Log every non-obvious decision.** "I chose Zustand over Redux because the team's other repo uses it and the cognitive cost was lower" is worth logging. "I added a button" is not.

### 2.7. Commits — conventional, atomic

```
<type>(<scope>): <subject>

<optional body>

Co-Authored-By: Claude Opus 4.7 (1M context) <noreply@anthropic.com>
```

`type` ∈ `feat | fix | refactor | test | docs | chore | perf | ci | style`
`scope` ∈ `auth | users | posts | api | ui | shared | ci | docker | <pack-or-slice-name>`

**Atomic = one concern per commit.** Fixing a bug + adding a feature = two commits. Tests for a feature commit alongside the feature (or as a preceding "test:" commit if you wrote them first per TDD §2.2).

### 3.1. Subagents

Specialized agents at `.claude/agents/*.md`. Invoke via the `Agent` tool with `subagent_type: <name>`. Each agent has its own context window and tool allowlist — use them to keep the main context clean and to delegate work that benefits from a fresh perspective.

| Agent | When to call it |
|---|---|
| `architect` | **Before** executing a non-trivial plan. Validates against Packwerk + FSD layer rules. Returns PASS or VIOLATIONS list. |
| `tester` | TDD-Red phase. Generates RSpec/Vitest/Playwright files that fail for the right reason, then runs them to confirm Red. |
| `reviewer` | After implementing, before committing. Reads `git diff --staged`, returns READY / CHANGES REQUESTED / BLOCKER with §5 violations highlighted. |
| `migrator` | For any schema change. Generates the migration following strong_migrations rules and reports downtime risk. |
| `debugger` | When a test is failing or a bug is unclear. Applies the scientific method, returns a diagnosis (NOT a fix to apply). |

Delegating to a subagent is preferred over expanding the main thread when the task is well-scoped (one of the rows above). The main Claude pulls in the agent's report and acts on it.

## 5. Anti-patterns — DO NOT DO

| ❌ Anti-pattern | ✅ Do this instead |
|---|---|
| Skip tests for "small" changes | Write at least one assertion. Always. |
| Put domain logic in `app/services/` (root) | Put it in a pack: `packs/<domain>/app/...` |
| Import across FSD sibling slices | Refactor the shared logic down to `entities/` or `shared/` |
| Deep import `@/features/auth-by-email/ui/SignInForm` | Import from `@/features/auth-by-email` (its `index.ts`) |
| Store JWT in `localStorage` | httpOnly signed cookie (already wired in `Api::V1::AuthController`) |
| Use `useState` for cross-component data | Zustand for client state, TanStack Query for server state |
| Catch and swallow errors | Re-raise (backend) or surface in UI via `toApiError` (frontend) |
| Hand-write a JSON shape twice | Generate once: rswag → `make typegen` → use `types.gen.ts` |
| Commit with `--no-verify` | Fix the hook failure |
| Bypass Pundit with conditional logic in controller | Add a policy method |
| Use `any` in TypeScript | Type it. If truly unknown, `unknown` + narrow. |
| Hard-code config in code | `ENV.fetch("VAR", default)` (backend), `env.ts` (frontend) |
| Write a UI component from scratch | Try `magic-mcp` first (§2.1) |
| Answer a library question from memory | Use `context7` first (§2.1) |

Order of escalation:

1. Re-read the relevant section of this file.
2. Read the latest log in `.claude/logs/` — was this discussed?
3. **Check `docs/TROUBLESHOOTING.md`** — known pain points for this stack are documented there.

5. Read existing similar code (search for analogous pack/slice).
6. **Try a subagent (§3.1)**: `debugger` for failing tests, `architect` for layer questions, `reviewer` for "is this code good?".
7. Ask the user a focused question via `AskUserQuestion`. Multiple options preferred over a vague open-ended ask.
8. If user is unavailable, propose a plan in plan-mode with the trade-offs visible.

## 7. Examples — concrete walkthroughs

### Example A. User says: "Add comments to posts"

1. Plan mode ON.
2. `context7` lookup: confirm latest Rails associations syntax, latest TanStack Query mutation API.
3. Read: `packs/posts/app/models/post.rb`, `packs/api/app/controllers/api/v1/posts_controller.rb`, `frontend/src/entities/post/`.
4. Plan:
   - **New backend pack `comments`** in `business_domain` layer (depends on `users`, `posts`, `shared`).
   - Migration: `comments` table (post_id, author_id, body).
   - Model: `Comment` belongs_to `:post`, `:author`.
   - Service: `Comments::Create` (Result-pattern, validates post is published or owned).
   - Policy: `CommentPolicy` (anyone authenticated can create on visible post, only author can delete).
   - Controller: `Api::V1::CommentsController` (nested under `/posts/:post_id/comments`).
   - Specs: model, service, policy, request — written FIRST.
   - **New frontend slice** `entities/comment` (types, query, CommentCard).
   - **New feature** `features/create-comment` (form + mutation + magic-mcp for the UI).
   - Update `widgets/post-feed` to optionally load comments per post (lazy).
5. ExitPlanMode → user approves.
6. Execute test-first per §2.2.
7. `/check-all`. Green.
8. `make log`, fill, commit.

### Example B. User says: "Make the post card prettier"

1. **No plan mode** if the change is self-contained in `entities/post/ui/PostCard.tsx`.
2. **`magic-mcp` lookup** — generate inspiration for blog-post cards.
3. Pick one. Adapt to our `Card`/`CardHeader` from `@/shared/ui` (don't re-implement primitives).
4. Update `Button.test.tsx` and `PostCard.test.tsx` if any markup changed in a way tests depended on.
5. `pnpm test --run` + `pnpm lint`.
6. Commit `style(posts): polish PostCard`.

### Example C. User says: "Library X is broken in our setup"

1. **`context7`** on library X first thing.
2. Compare what docs say vs. our actual config (`Gemfile`, `package.json`, configs).
3. Reproduce locally. Add a **failing test** that reproduces (TDD-style; use VCR / MSW if external).
4. Fix.
5. Test goes green. Add log entry tagged `fix(<scope>): <library X> ...`.
